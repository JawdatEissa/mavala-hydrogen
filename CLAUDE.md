## Project Overview

Custom **Remix (Vite) + TypeScript** storefront for the Mavala beauty brand. Despite the repo name (`mavala-hydrogen`), this does **not** use the `@shopify/hydrogen` npm package — it integrates directly with the Shopify Storefront API from server-side Remix loaders.

Deployed on **Vercel**.

## Shopify Dev MCP (required workflow)

The **Shopify Dev MCP** server (`@shopify/dev-mcp`) is configured for this project. Before writing any Shopify-related script, uploading files/media to Shopify, or making Admin/Storefront API calls:

1. **Inspect the prompt** — understand what the user is asking and which Shopify APIs are involved.
2. **Consult the Shopify Dev MCP** — look up the relevant API documentation, GraphQL mutations/queries, and schemas through the MCP tools. Verify field names, input types, and required scopes against the official docs.
3. **Then act** — only after confirming the correct API shape, write the script or provide guidance to the user.

Do **not** guess at Shopify GraphQL mutations or API behavior from memory. Always use the MCP as the source of truth for Shopify API details.

## Architecture

### Data sources (two-tier)
Product data comes from two sources merged at runtime in loaders:

1. **Live Shopify Storefront API** (`app/lib/shopify-storefront.server.ts`) — primary source, queried by product handle.
2. **Bundled JSON catalog** (`app/data/all-products.json`, `app/data/image-manifest.json`) — fallback and enrichment layer. Includes local images, shade data, descriptions, and page-specific data JSONs.

The PDP loader (`app/routes/products.$handle.tsx`) tries Shopify first, merges local data via `enrichShopifyProductFromJson()` if the same handle exists in JSON, then falls back to JSON-only if Shopify has no match.

### Server-side module convention
Files named `*.server.ts` are server-only. Key modules:
- `app/lib/shopify-storefront.server.ts` — Storefront GraphQL queries, product fetching
- `app/lib/shopify-admin.server.ts` — Admin API with in-memory token cache (client credentials flow, tokens refresh ~60s before expiry)
- `app/lib/shopify-cart.server.ts` — Cart CRUD via Storefront API; cart ID stored in httpOnly cookie `mavala_shopify_cart_id`
- `app/lib/openai.server.ts` — OpenAI embeddings + chat completions (RAG for AI chatbot)
- `app/lib/supabase.server.ts` — Supabase pgvector for chunk retrieval and semantic cache
- `app/lib/auth.server.ts` — Customer session and rate-limit helpers
- `app/lib/scraped-products.server.ts` — Loads and caches `all-products.json`; resolves local images from manifest; `ScrapedProduct` is the canonical product type

### AI Chatbot (RAG pipeline)
Route: `app/routes/api.chat.tsx` (Remix action endpoint)

Flow: rate limit → semantic cache lookup (Supabase) → embed question (OpenAI `text-embedding-3-small`) → retrieve chunks (Supabase pgvector) → chat completion (GPT-4o primary, fallback to GPT-4o-mini) → parse `<<<PRODUCTS:[...]>>>` marker from LLM response → cache result.

The LLM is instructed to append `<<<PRODUCTS:["handle1","handle2"]>>>` at the end of every response; `parseLLMRecommendedProducts()` strips this and surfaces the handles to the frontend as product cards.

Knowledge ingestion: `scripts/generate-product-knowledge.ts` → `scripts/ingest-content.ts` → Supabase.

### Color/shade pages
`app/routes/color.tsx` consumes `app/data/color_mapping_*.json` files (one per shade collection) and `app/data/shade_colors.json`. The `ShadeDrawer` component (`app/components/ShadeDrawer.tsx`) shows individual shade swatches from the image manifest.

### Image manifest
`app/data/image-manifest.json` maps product slugs and shade folder names to local image paths under `public/`. Generated by `scripts/generate-image-manifest.mjs`. Must exist before build; `npm run build` handles this automatically.

### Category/page data
Static page content (hero images, descriptions, featured products) lives in `app/data/*-page.json` files consumed by page-specific routes.

## Key Patterns

- **Product handle** is the shared key between Shopify and local JSON data — a mismatch means no enrichment.
- Products with `product.shopify.defaultVariantGid` can be added to cart; JSON-only products show "not available for online checkout."
- The `ScrapedProduct` type (in `scraped-products.server.ts`) is the normalized shape used throughout the UI — Shopify responses are mapped into this shape via `mapStorefrontProductToScrapedProduct()`.
- Category filtering in `getProductsByCategory()` first matches the `categories` array, then falls back to keyword matching against title/slug/description.
- Auth gating for the chatbot is currently disabled (`REQUIRE_AUTH_IN_PRODUCTION = false` in `api.chat.tsx`).

## Product Image Display Rules

### Special grid layout (`products.$handle.tsx` → `ImageGallery`)

Products that need a non-standard image grid are identified by a `useSpecialGridLayout` boolean:
- `mavala-scientifique-1`, `mavala-scientifique-k`, `mavala-stop`, `nailactan-1`
- All slugs ending in `-shades` or `-colors` (e.g. `pink-shades`, `pearl-colors`)

**Critical rule:** Use `object-contain` for secondary images that are standalone product shots (bottle with label). `object-cover` is only acceptable for lifestyle/marketing images where intentional cropping is fine.

- `-shades` / `-colors` slugs: secondary images use `object-cover object-top` with `maxHeight: 589px` (crops from bottom, keeping cap/label visible)
- **Exception — `bio-colors`**: stays in special grid but uses `object-contain` (individual bottle shots must not be cropped)

The relevant conditional in `ImageGallery`: `isBioColors ? "object-contain" : isShadeCollection ? "object-cover object-top" : ...`

### Per-product image scale/offset adjustments

Two product card components render products — adjustments must be made in **both** if the product appears on the Color page:

| Component | File | Used on |
|---|---|---|
| `ProductCard` | `app/components/ProductCard.tsx` | Most listing pages |
| `ColorProductCard` | `app/routes/color.tsx` | Color page only |

Each has two lookup objects keyed by slug:
- `imageScaleAdjustments` — scale multiplier (`1.0` = no change; do **not** use `0`, it's falsy)
- `imageOffsetAdjustments` — `translateY` string (e.g. `"-6.5%"` moves up, `"6.5%"` moves down)

## Shopify Sync Scripts

The sync scripts are **idempotent** — re-running updates existing products where the handle already exists.

- Catalog sync handle = product `slug` from `all-products.json`
- Shade sync handle = `mavala-shade-{normalized-name}` (e.g. `1 ANKARA` → `mavala-shade-1-ankara`)

Useful CLI flags (pass after `--` when using npm scripts):

| Flag | Effect |
|---|---|
| `--dry-run` | Parse and log only; no API calls |
| `--limit N` | Process only first N products (useful for spot-checks) |
| `--delay MS` | Wait MS ms between products; use `--delay 500` to avoid throttling |
| `--skip-publish` | Skip publishing to sales channels |

After syncing, products are created in Admin but **not visible on the Storefront** until published to the correct sales channels. If `read_publications`/`write_publications` scopes are missing, publish manually in Shopify Admin.

Re-running shades may **append duplicate media** in Shopify if the same product was previously synced — remove extras once in Admin if this happens.

## AI Chatbot Configuration

- Rate limit: 20 requests/minute per user — configured in `app/lib/chat.types.ts` (`RATE_LIMIT_WINDOW_MS`, `RATE_LIMIT_MAX_CALLS`)
- Semantic cache hit threshold: 90% cosine similarity; retrieves top 6 chunks
- System prompt: `SYSTEM_PROMPT` constant in `app/lib/openai.server.ts`
- Supabase DB schema: `scripts/setup-database.sql` — run this in the Supabase SQL editor to initialize pgvector tables/functions

Verify chunk ingestion in Supabase:
```sql
SELECT source, count(*) FROM chunks GROUP BY source;
```
Expected: `blog` ~20–50, `product` ~200+, `quiz` ~15.
